# SOP-015 — System File Change Proposal Procedure

**Owner:** Larry
**Applies to:** All specialists
**Trigger:** Any proposed change to a GL, SOP, AGENT.md, or CLAUDE.md
**Last updated:** 2026-06-18

---

## Procedure

### Step 1 — Prepare

Larry produces a complete proposal: what changes, in which file, and why. The proposal is complete before it reaches anyone. No open questions. No missing information.

### Step 2 — Iris review (when applicable)

Iris reviews the proposal when it touches:
- Any GL or SOP file
- CLAUDE.md
- Governance structure (approval gates, escalation rules, agent ownership)

Iris does **not** review AGENT.md-only changes. Those go directly to the owner.

Iris checks: is the proposal consistent with existing governance, is anything missing, are there risks or hard boundaries. Iris advises. She does not authorize.

If Iris is not available, Larry states this and presents without Iris review.

### Step 3 — Present to owner

Larry presents: the proposed change + Iris's assessment (if applicable). One proposal, one decision. The owner's only required response is yes, no, or a correction.

### Step 4 — Execute

Larry executes only after explicit owner confirmation. No exceptions.

---

## Hard rules

- No file is written before Step 4
- Iris review is advisory — owner confirmation is always required regardless
- Owner's "yes" covers the proposal as presented — any deviation requires a new proposal
- Changelog entry required in every modified system file: date, what changed, who

---

## Deliverable lifecycle (one rule)

Finished deliverables move to `Deliverables/Archive/` when the work is done. Knowledge worth keeping — a lesson, a decision, a reference — gets extracted to its domain (PKM, session-logs, BKML). The owner decides what gets extracted. Nothing is extracted automatically.
