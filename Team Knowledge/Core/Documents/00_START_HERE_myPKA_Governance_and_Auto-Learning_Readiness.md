## 5. Future Ideas to Preserve

The following candidates are recorded here as forward references. None are approved for implementation. None are backlog items unless Owner explicitly approves backlog creation. No action may be taken on any candidate without a new proposal and explicit Owner approval.

---

### A. Governance Gatekeeper / Owner Review Advisor

**Purpose:** Independent process quality gate that helps Owner Walter Kamer verify route, DP status, Review Gate readiness, lifecycle readiness, missing files, and exact next steps before a governance action proceeds.

**Why it matters:** During the SOP-017 amendment flow, the Owner needed to manually verify route classification, DP sequence, and lifecycle packet correctness across multiple turns. A dedicated role with structured checklists would surface gaps earlier and reduce review burden on the Owner.

**Status:** Future workstream candidate only. Not approved for implementation. Not a backlog item unless Owner explicitly approves backlog creation later.

---

### B. Auto-Learning Governance and Processing Flow

**Purpose:** Define how learning candidates are detected, classified, routed, approved, processed, reported, archived, and cleaned up across all knowledge types: PKM, BKM (Kamer E-commerce and Geldstroom Regie), Core governance knowledge, project and workstream notes, decisions, lessons learned, references, backlog candidates, and archive handling.

**Why it matters:** The current governance system has strong proposal and review flows but no defined end-to-end flow for detecting and processing learning candidates automatically. Each candidate currently requires manual identification and routing.

**Must include:**
- Detection triggers (what events produce learning candidates)
- Classification logic (which knowledge type, which domain, which destination)
- Routing rules (which specialist processes which type)
- Approval gates (what requires Owner review before processing)
- Processing procedures (how each knowledge type is extracted and written)
- Reporting and archiving requirements
- Cleanup and source folder closure rules

**Status:** Future workstream candidate only. Not approved for implementation. Not a backlog item unless Owner explicitly approves backlog creation later.

---

### C. Codex Controlled Pilot

**Purpose:** Evaluate Codex as an additional execution engine for the myPKA AI team, specifically for tasks that benefit from autonomous multi-step execution within a governed scope.

**Sequencing constraint:** Must not be started until Governance Gatekeeper and Auto-Learning Governance are clarified. Codex without a functioning quality gate and learning flow would bypass the governance structure currently being built.

**Status:** Future candidate only. Not approved for implementation. Evaluation may not begin until candidates A and B are defined.

---

### D. Strict File-Based Governance Artifact Flow for Auto-Detection and Auto-Processing

**Purpose:** Define how proposal files, RCP files, Review Gate findings, DP decision records, implementation reports, lifecycle decision packets, lifecycle execution reports, and cleanup reports are detected, validated, processed, and archived — with durable file-based state at each governance step.

**Specific process lessons to evaluate later:**
- Review Gate findings should normally be saved as a findings file unless Owner explicitly approves chat-only findings.
- DP decision records should be file-based where they control later steps.
- DP-4 decision recording should preferably be separated from implementation execution in future process design.
- Implementation runs should read and validate a pre-existing DP-4 decision record instead of creating it in the same run.
- Current workflow execution and future auto-processing design should not be mixed in the same active governance flow.

**Source:** Preserved from the SOP-017 lifecycle execution report — `Deliverables/20260605_Core_SOP-017 Amendment Lifecycle Execution/lifecycle-execution-report-sop-017-source-folder-closure-amendment-v02.md` Section 5 (R4 Governance reference, Indexed marker).

**Status:** Future candidate only. Not approved for implementation. Not a backlog item unless Owner explicitly approves backlog creation later.
