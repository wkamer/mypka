# Action Report 10 — Process Learning: Hiring Ping-Pong

**Captured by:** Larry, Orchestrator

**Status:** Triage proposal — awaiting Owner decision before any file changes.

---

## 1. Learning Summary

The Iris hire flow became unnecessarily fragmented. Governance was applied at every intermediate step — including thinking, drafting, and preparation steps that carried no structural risk — instead of being reserved for actions that are destructive or hard to reverse. This created high Owner load, slow progress, and repeated ping-pong between Owner and Nolan for steps Nolan already owns under SOP-003.

---

## 2. What Happened

Nolan was micro-directed through a long sequence of small governance checkpoints:
- Clarifying questions (action-report-06d)
- Overlap check (action-report-06e)
- Initial AGENT.md draft — Owner review gate
- Revised AGENT.md draft — Owner review gate
- Final AGENT.md draft — Owner review gate
- Step 3: folder creation — Owner review gate
- Step 4: agent-index.md update — Owner review gate

Each step was individually routed through Larry, confirmed by Owner, and only then executed. Some of those gates were appropriate (folder creation, file writes, index updates). Others were intermediate preparation steps where Nolan already had the scope, source material, and SOP authority to proceed without a checkpoint.

---

## 3. Why It Matters

Two failure modes compound each other:

**Failure mode 1 — Governance at the wrong level.** Governance exists to protect against destructive, structural, or hard-to-reverse actions. When it is applied to every reasoning or drafting step, it stops being a quality gate and becomes a bottleneck. Owner attention is finite. Spending it on "did Nolan ask the right clarifying questions" reduces capacity for "is this AGENT.md correct."

**Failure mode 2 — Nolan not owning his SOP.** SOP-003 exists precisely so Nolan can execute a hire autonomously up to a defined boundary. When every sub-step requires explicit Owner approval, Nolan is not following SOP-003 — he is executing a manual checklist dictated by Larry. This does not build a reliable, autonomous HR process. It recreates the same Owner load every hire.

The combined effect: a single hire with a well-defined scope took a large number of turns that should have been compressed into 3 to 4 Owner decisions.

---

## 4. Recommended Destination

Triage across the five candidate destinations:

**a. Nolan AGENT.md behavioral guidance — YES, primary destination**
Nolan needs an explicit behavioral rule: follow SOP-003 autonomously within the non-destructive sandbox. Do not surface intermediate steps. Surface only at defined implementation boundaries. This is agent behavior — it belongs in his AGENT.md.

**b. SOP-003 hiring process refinement — YES, secondary destination**
SOP-003 should explicitly define the implementation boundary gates — the moments where Owner confirmation is required before proceeding. Anything before those gates is Nolan's autonomous territory. The current SOP may not draw this line clearly enough.

**c. Governance guideline / lean governance principle — OPTIONAL**
A lean governance principle ("governance gates structural actions, not reasoning steps") could be added to GL-019 or a new GL if this pattern recurs across other processes. Not urgent for this hire. Flag as a candidate if the same pattern appears in a second workflow.

**d. Backlog improvement candidate — NO**
This is not a backlog item. It is an active behavioral correction that should be applied to the next hire. Putting it in a backlog risks it being forgotten.

**e. Session learning only — NO**
A session-learning-only record does not change Nolan's behavior or SOP-003. The learning would need to be re-explained on the next hire. Not durable enough.
