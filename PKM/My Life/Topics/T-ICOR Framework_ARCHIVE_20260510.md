## Wat is ICOR?

ICOR is een persoonlijk en team productiviteitsframework van Paperless Movement. De naam staat voor **Input → Control → Output → Result** — een complete methodologie voor persoonlijke en zakelijke productiviteit. De praktische kern is het bouwen van een functionerend systeem dat van idee naar actie gaat.

Sienna gebruikt het ICOR Framework als leidend principe voor persoonlijke begeleiding van de eigenaar.

### De ICOR Journey

| Letter | Naam | Definitie |
|---|---|---|
| **I** | Input | Alles begint hier. Één plek waar nieuwe inputs landen voor ze verspreid raken over je leven. |
| **C** | Control | Ruwe inputs omzetten naar bruikbare notities, taken en referenties. Niets gaat verloren. Niets blijft vaag. |
| **O** | Output | Elk item plaatsen waar het hoort via de vier kwadranten. Je digitale leven krijgt een heldere structuur die werkt met elk tool. |
| **R** | Refine | Regelmatige check-ins houden je systeem levend. Een wekelijkse reset die minuten kost, geen uren. |

### Core Mental Models (overzicht)

| Model | Kern |
|---|---|
| The Capturing Beast | 3-filter model voor intentionele informatieverwerving |
| The Execution Beast | Weekplanning met doelalignering en balans |
| The Thought Lab | Gestructureerd diep denken voor compound kennis |
| PEA (Plan/Execute/Align) | Strategische alignering in dagelijks werk bewaken |
| Signal vs Noise | Waardevolle informatie filteren van afleiding |

## Kernhiërarchie

Goals → Projects → Workstreams → Taken

Prioriteit van taken wordt nooit handmatig bepaald. De positie in de hiërarchie bepaalt de urgentie. Tijdens de daily planning krijgen relevante taken een Due Date — dat is het planningssignaal.

## Goal-Setting Workflow

**De constraint**: Een Goal kan alleen bereikt worden via een Project of een Habit. Er is geen derde vorm.

### De 3 stappen

**Stap 1: Anchor — Naam de bestemming**
Schrijf de bereikte staat in één zin met een getal. "10 kilo afvallen voor Q3." Als je zou weten dat je het bereikt hebt, is het een bestemming.

**Stap 2: Anchor to Key Element — Permanente grond**
Geen Key Element-anker = geen Goal. Promoveer het Topic naar Key Element, of laat de wens los.

**Stap 3: Decompose — In vormen**
Noem het Project (eenmalige inspanning met start en einde) en/of de Habit (terugkerende cadence) die de Goal draagt. Nooit geen van beide.

### Kwartaalritme

Set → Execute → Review → Rotate

- **Orphan Goals** (geen Project of Habit) → decomponeren of loslaten
- **Driftende executie** (taken die niet aan een goal hangen) → herverbinden of sluiten

## PEA — Plan / Execute / Align

De meeste mensen zitten vast in de Execute-loop: taken doen zonder alignering te controleren. PEA doorbreekt dit.

| Fase | Definitie |
|---|---|
| **Plan** | Definieer wat er toe doet (goals, prioriteiten, uitkomsten) |
| **Execute** | Doe het werk (taken, processen, acties) |
| **Align** | Controleer of Execute de Plan dient — dit is wat de meeste mensen overslaan |

### Output Elements Hiërarchie

**Goals → Projects → Workstreams → Operations → Tasks**

Als taken niet opwaarts aligneren, ben je druk maar niet productief.

### PEA Check — structuur

1. **Plan**: Wat zijn je top 3 goals nu?
2. **Execute**: Wat heb je deze week gedaan (of gepland)?
3. **Align**: Kan je elke taak terugvoeren naar een goal?

| Taak | → Project | → Goal | Aligned? |
|---|---|---|---|
| [Taak] | [Project?] | [Goal?] | Ja / Nee / Deels |

**Alignment Score** = aantal Ja / totaal taken = percentage

4. **Correct**:
   - Taken zonder goaldkoppeling → elimineren, delegeren of batchen?
   - Goals zonder taken → worden ze verwaarloosd?
   - Over-geïnvesteerde goals → te veel tijd op één goal ten koste van andere?

## SOP-structuur (Standard Operating Procedure)

In de ICOR-hiërarchie documenteert een SOP één enkel **Process** in genoeg detail dat iedereen het kan volgen. SOPs bevatten BPMN-geïnspireerde elementen.

### SOP-template

```markdown
## SOP: [Procesnaam]
Workstream: [naam] | Workflow: [naam]
Owner: [rol] | Frequency: [cadentie]
Last updated: [datum] | Version: 1.0

### Trigger
[Welk event start dit process]

### Prerequisites
- [ ] [Wat moet waar zijn voor je begint]
- [ ] [Benodigde toegang/tools/informatie]

### Steps
1. **[Werkwoord] [wat]**
   - Tool: [welke tool]
   - Details: [specifieke instructies]
   - Expected result: [hoe succes eruitziet]

### End State
[Hoe het voltooide process eruitziet]

### Automation Potential
- Current type: [Manual / Template-assisted / Automated / AI-empowered]
- Target type: [aanbevolen niveau]
- How: [korte automatiseringsaanpak]

### Common Issues
- [Probleem] → [Oplossing]
```

## Workstream → Workflow → Process Hiërarchie

### Definitie per niveau

| Niveau | Definitie | Voorbeeld |
|---|---|---|
| **Workstream** | Terugkerend operationeel gebied | "Content Marketing", "Klant Onboarding" |
| **Workflow** | Reeks stappen binnen een Workstream | "Blog Post Publiceren" |
| **Process** | Individuele taak binnen een Workflow | "Headline-opties schrijven" |

### Automatiseringstype per Process

| Type | Definitie |
|---|---|
| **Manual** | Vereist menselijke inspanning elke keer |
| **Automated** | Afgehandeld door software (scripts, integraties) |
| **AI-empowered** | AI vergroot of vervangt menselijke inspanning |

### Workstream-documentatie structuur

Voor elke Workstream documenteer je:
- **Owner** — wie is verantwoordelijk
- **Frequency** — hoe vaak draait dit (dagelijks, wekelijks, per project)
- **Goal** — welke uitkomst produceert deze Workstream
- **Workflows** — per workflow een tabel met Process, Type, Tool, Duur
- **Automation Summary** — % Manual / Automated / AI-empowered
- **Optimalisatiekansen** — welke processen kunnen geautomatiseerd of AI-empowered worden

## Thought Lab — Diep Denken

### Context

| Type | Definitie |
|---|---|
| **Shallow Thinking** | Reactief, oppervlakkig, consumeren zonder verwerken. Lezen zonder reflecteren. |
| **Deep Thinking** | Intentioneel, verbindend, generatief. Nieuwe inzichten creëren uit bestaande kennis. |

De **Thought Lab** is je vaste mentale ruimte voor Deep Thinking.

### Sessiestructuur (3 fasen)

**Fase 1: Diverge (10 min)**
Genereer zo veel mogelijk verbindingen:
- Hoe relateert dit aan huidige projecten?
- Welke mental models zijn van toepassing?
- Hoe ziet het tegenovergestelde perspectief eruit?
- Waar heb ik vergelijkbare patronen eerder gezien?

**Fase 2: Converge (10 min)**
Destilleer de meest waardevolle inzichten:
- Welke verbindingen verrasten je?
- Wat is het kerninsicht dat je aan iemand anders zou uitleggen?
- Hoe verandert dit je begrip?

**Fase 3: Compound (5 min)**
Verbind met je bestaande kennissysteem:
- Welke bestaande notities versterkt of daagt dit uit?
- Welke nieuwe vragen roept dit op?
- Waar in je PKM-systeem leeft dit inzicht?

### Compound Knowledge Effect

Elke Thought Lab-sessie maakt bestaande kennis waardevoller door verbindingen te leggen die er eerder niet waren.

## The Execution Beast — Weekplanning

### Kernprincipes

| Principe | Uitleg |
|---|---|
| **Weekly Goals** | Sluiten aan op kwartaal-/maanddoelen (top-down) |
| **Brain Zen** | De kalme zekerheid van precies weten wat er gepland staat |
| **Balance** | Plan 4–6 uur diep productief werk per dag, niet 8+ |
| **Buffer** | Laat 30% ongescheduled voor onverwachte taken |
| **Goal Alignment** | Elke taak is terug te herleiden naar een goal |

### Weekplanningsstructuur

1. **Review vorige week** — Wins, rollovers, verrassingen
2. **Goal alignment check** — Wat is de sleutelvooruitgang per kwartaaldoel deze week?
3. **Weekdoelen bepalen** — 3–5 weekly goals met prioriteit en uurschatting
4. **Balance check**:
   - Totaal geschatte uren vs beschikbare productieve uren (4–6/dag × 5 = 20–30u)
   - Buffer (30%) = 6–9u gereserveerd
   - Netto beschikbaar: 14–21u
5. **Brain Zen setup** — Highlight of the Day per dag bepalen

## Daily Planning — Highlight of the Day

### Kernconcepten

| Concept | Definitie |
|---|---|
| **Highlight of the Day** | DE ÉÉN taak die de dag tot een succes maakt als hij klaar is |
| **Deep Work** | Gefocust, cognitief veeleisend werk dat waarde creëert (schrijven, coderen, ontwerpen, strategiseren) |
| **Shallow Work** | Administratie, e-mail, berichten, routineupdates |
| **Sequentiality** | Één ding tegelijk — multitasking vernietigt Deep Work |
| **Time Blocking** | Specifieke tijdsloten toewijzen aan specifieke taken |
| **Batching** | Gelijksoortige Shallow Work taken groeperen (alle e-mails tegelijk, alle berichten tegelijk) |

### Dagplanningsstructuur

1. **Kies je Highlight** — Welke ONE taak maakt de dag tot een win?
2. **Categoriseer resterende taken** — Deep of Shallow, duur, prioriteit
3. **Time block je dag**:
   - Deep Work blok (bescherm dit ten koste van alles)
   - Shallow Work batch (gegroepeerd)
   - Buffer (open laten voor verrassingen)

### Dagregels

- Geen context-switching tijdens Deep Work blokken
- Berichten en e-mail in één gebundeld tijdslot
- Als de Highlight klaar is, is de dag een succes

## Signal vs Noise

### Definities

- **Signal**: Informatie die direct jouw goals, projecten of key topics dient
- **Noise**: Al het andere — interessant maar nu niet relevant
- **Inbox**: Tijdelijke opvang voor nieuwe informatie (nooit permanent)
- **Bucket**: Georganiseerde opslag voor verwerkte informatie (permanente referentie)

### Bronclassificatie

| Type | Definitie | Actie |
|---|---|---|
| **Strong Signal** | Helpt direct mijn top 3 prioriteiten | Bewaren, routen naar Inbox |
| **Weak Signal** | Soms relevant | Wekelijks gebundeld verwerken |
| **Noise** | Zelden relevant voor mijn werk | Uitschrijven / muten / verwijderen |

### Inbox-regels

- **Minimaal** — zo weinig mogelijk inboxen (ideaal 1–3)
- **Tijdelijk** — niets blijft langer dan 1 week in de Inbox
- **Regelmatig verwerkt** — vaste verwerkingstijd inplannen

## PKM Knowledge Cycle

Informatie doorloopt vijf stadia. Elke stap voegt waarde toe:

**DISCOVER → CAPTURE → PROCESS → ACTION → SHARE**

| Stadium | Wat er gebeurt |
|---|---|
| **DISCOVER** | Informatie opmerken — hoe informatie jou vindt en hoe jij informatie vindt |
| **CAPTURE** | De brug tussen iets nuttigs opmerken en het veilig in je systeem hebben — in onder 5 seconden |
| **PROCESS** | Beslissen wat diepere investering verdient via The Thought Lab — 4-vragen evaluatie |
| **ACTION** | Twee paden: heldere taak → task management systeem, of complex idee → Deep Thinking omgeving |
| **SHARE** | Persoonlijke kennis wordt collectieve kennis — van PKM naar BKM of publieke output |

### Stage 5: SHARE — detail

Niet alles moet gedeeld worden. De meeste verwerkte kennis blijft privé. Deel alleen wat anderen oprecht ten goede komt.

**Vraag:** Is het de moeite waard om te delen?
- **JA** → Personal World → Business World: team, project management systeem, The Idea Incubator, of publiek (Outer)
- **NEE** → blijft in Shallow Thinking System (Personal World)

Delen stress-test je ideeën aan andere perspectieven en onthult blinde vlekken die je alleen niet kunt zien. Dit is waar PKM BKM ontmoet: persoonlijke inzichten worden teamdocumentatie, voorstellen of gepubliceerde content.

**Delen voltooit de cyclus en herstart hem:** feedback van anderen creëert nieuwe informatie om te ontdekken.

### Stage 4: ACTION — detail

Twee paden:

**Pad 1: Je weet wat je moet doen**
Schrijf een concrete volgende stap en verplaats naar PPM (persoonlijk) of BPM (team/zakelijk).

**Pad 2: Je hebt meer denktijd nodig**
Verplaats naar je Deep Thinking omgeving voor verdere verkenning.

Dit is waar PKM verbindt met je taakbeheersysteem. Kennis blijft in PKM, acties gaan eruit.

> Informatie die nooit een beslissing, taak of deliverable wordt, is slechts goed georganiseerde opslag.

### Stage 3: PROCESS — detail

The Thought Lab evalueert gevangen items via 4 vragen:

1. Verbetert dit mijn kennis?
2. Is het het verkennen waard?
3. Heeft het impact op mijn projecten?
4. Zijn er duidelijke volgende stappen?

**JA op één vraag** → naar Action (Deep Thinking System).
**NEE op alle vier** → blijft in Shallow Thinking System.

Wacht 1–2 dagen voor je verwerkt. Frisse ogen scheiden echte waarde van initiële opwinding. De meeste gevangen items hebben geen diepe aandacht nodig. Dat is het systeem dat correct werkt, geen fout.

### Stage 2: CAPTURE — detail

Twee inboxen:
- **Outer World** — artikelen, links, rapporten (van buiten)
- **Inner World** — eigen ideeën, reflecties, observaties (van binnen)

Werkwijze: vang het item, voeg minimale context toe (bron, waarom het er toe doet, één relevante bucket). Snelheid is alles — als je capture-tool meer dan 5 seconden kost, sla je het over onder druk.

**Capture is geen processing.** Sla nu op, evalueer later tijdens je dagelijkse inbox-clearing. Inbox → Noise (prullenbak) of Signal (juiste Bucket in PKM-systeem).

### Stage 1: DISCOVER — detail

Twee modi:
- **Intentional discovery** — gericht zoeken met doel. Gefocust en efficiënt.
- **Accidental discovery** — toevallig iets nuttigs tegenkomen. Vereist de Capturing Beast als schild.

Ongecontroleerd browsen verbrandt tijd aan content die nergens mee verbindt. Herken het patroon en stop.

**Enige taak bij DISCOVER: informatie opmerken.** Vangen en verwerken gebeuren in latere stadia.

### Kwaliteitscheck bij DISCOVER

- **Relevantie** (1–5): Hoe relevant voor huidige goals?
- **Nieuwheid** (1–5): Hoe nieuw is dit voor mij?
- Beide onder 3 → overwegen te verwerpen.

## REFINE — Automatiseringsniveaus

ICOR beschrijft vijf niveaus van automatisering, van minst naar meest geautomatiseerd:

| Niveau | Omschrijving |
|---|---|
| 1. Manual | Mens doet het elke keer, from scratch |
| 2. Template-assisted | Mens volgt een template of checklist |
| 3. Software Automated | Software handelt het af met minimale input |
| 4. AI-empowered | AI handelt het af met menselijk toezicht |
| 5. Running on Autopilot | Volledig geautomatiseerd, draait zonder interventie |

### Automatisering Prioriteitsmatrix

| Categorie | ROI | Inspanning | Actie |
|---|---|---|---|
| **Quick Wins** | Hoog | Laag | Deze week automatiseren |
| **Strategic Projects** | Hoog | Hoog | Plannen voor volgende maand |
| **Nice-to-Have** | Laag | Laag | Doen als er tijd is |
| **Skip** | Laag | Hoog | Niet de moeite waard |

### ROI Formule

**Automation ROI** = (tijdsbesparing × frequentie) ÷ setup-inspanning

## Output Formula — Kwartaaldoelen

### Top-Down vs Bottom-Up

**Top-Down**: Visie → Goals → Projects → Workstreams → Operations
**Bottom-Up**: Huidige taken → welke workstreams bedienen ze → welke goals ondersteunen ze

De sweet spot: goals waar jouw top-down ambitie en bottom-up realiteit het over eens zijn.

### Output Elements Hiërarchie

1. **Goals** — gewenste uitkomsten (meetbaar, tijdgebonden)
2. **Projects** — afgebakende inspanningen om goals te bereiken (hebben start- en einddatum)
3. **Workstreams** — terugkerende operationele gebieden (doorlopend, geen einddatum)
4. **Operations** — dagelijkse taakuitvoering

### Kwartaalplanning — structuur

**Part 1: Top-Down** — Wat zijn 2–3 goals die deze kwartaal de meeste vooruitgang boeken richting de 12-maands visie?

**Part 2: Bottom-Up** — Waar besteed je nu de meeste tijd aan? Welke workstreams bedienen de top-down goals, welke zijn puur onderhoud?

**Part 3: Alignment Check** — Per workstream: dient het een goal? Keep / Reduce / Eliminate.

**Part 4: Resource Allocatie** — ~500 productieve uren per kwartaal (5u/dag × 5 dagen × ~20 weken). Verdeel over goals + onderhoud.

## The Capturing Beast

Mental model voor informatieverwerking. Wanneer nieuwe informatie binnenkomt, doorloop je drie filters om te beslissen of het de moeite waard is om te bewaren.

### De 4 Filters

1. **Projects Filter** — Dient dit een huidig project?
2. **Key Elements Filter** — Relateert dit aan een Key Element?
3. **Habits Filter** — Ondersteunt dit een actieve Habit?
4. **Topics Filter** — Is dit verbonden aan een Topic dat ik aan het verkennen ben?

Passeert de informatie minstens één filter → bewaren. Geen filter → loslaten. Geen schuldgevoel, geen "just in case" bookmarks.

**Verwacht 50–70% minder te capturen dan voorheen.** Minder volume = meer signal. Vervang FOMO met JOMO (Joy of Missing Out).

### Capturing Beast Setup — Regels

- **Current Projects**: max 5. Meer dan 5 = te veel actieve projecten.
- **Key Elements**: 3–7 kerngebieden waar je expertise in opbouwt.
- **Top 3 Topics**: precies 3. De beperking is intentioneel. Stabiel week-to-week, mag kwartaal wijzigen.

### Capturing Beast Card (template)

```markdown
## My Capturing Beast — [Datum]
*Review en update maandelijks*

### Filter 1: Current Projects
1. [Project]
2. [Project]
3. [Project]

### Filter 2: Key Elements
1. [Element]
2. [Element]
3. [Element]

### Filter 3: Top 3 Topics
1. [Topic]
2. [Topic]
3. [Topic]

### Decision Rule
Pass 1+ filter → Capture naar inbox
Pass 0 filters → Loslaten. Het is ruis.

### Monthly Review Date: [1e van volgende maand]
```

### Workflow: DISCOVER → CAPTURE → PROCESS

**Step 1: DISCOVER (Signal vs Noise)**
- Is dit **Signal** (waardevol, actionable) of **Noise** (afleidend, lage waarde)?
- Snelle check: Doet dit er over 2 weken nog toe?

**Step 2: CAPTURE (Apply the 3 Filters)**
Run door elk filter met jouw actuele projecten, key elements en topics.

**Step 3: ROUTE**
- **Inbox** — actionable of tijdgevoelig → snel verwerken
- **Bucket** — waardevol maar niet urgent → referentie opslag
- **Discard** — geen filter gepasseerd → loslaten

## De 4 Kwadranten — Tool Ecosystem

ICOR verdeelt tools in 4 kwadranten. Een gezond ecosysteem heeft minimaal één kerntool per kwadrant, minimale overlap, sterke integratie en niet te veel tools.

| Kwadrant | Wat het dekt |
|---|---|
| **PKM** — Personal Knowledge Management | Persoonlijke kennis, notities, journaling, CRM |
| **BKM** — Business Knowledge Management | Zakelijke kennis, processen, klantdata |
| **PPM** — Personal Process Management | Persoonlijke taken, agenda, gewoonten, projecten |
| **BPM** — Business Process Management | Zakelijke taken, workflows, operaties |

### Ecosystem Health — criteria

- **Coverage**: elk kwadrant heeft minstens één kerntool
- **Minimale overlap**: geen twee tools doen hetzelfde
- **Sterke integratie**: tools delen data automatisch
- **Juiste schaal**: niet te veel (complexiteit) of te weinig (gaten)

| Gmail | PPM | Core |

| SQLite DBs | BKM / PKM | Core |

## SSOT — Single Source of Truth

### Principes

- **SSOT**: Elk stuk informatie heeft precies één gezaghebbende plek.
- **Inbox**: Tijdelijke opvangplaats — niets blijft hier permanent.
- **Buckets**: Permanente, georganiseerde opslag — de SSOT per informatietype.
- **Context — Tracing**: Informatie terugvoeren naar de oorsprong.
- **Context — Integrating**: Tools verbinden zodat informatie automatisch doorstroomt zonder handmatig kopiëren.

### Buckets (SSOT per type)

| Informatietype | SSOT |
|---|---|
| Taken & acties | Todoist |
| Notities & kennis | myPKA vault (Obsidian / Claude Code) |
| Projecten | PKM/My Life/Projects/ + Todoist |
| Bestanden & documenten | PKM/Documents/ |
| Contacten & CRM | PKM/CRM/ |
| Agenda & planning | Google Calendar |

### Inbox Flow

- **Primary Inbox**: Team Inbox/ (per domein)
- **Processing**: Per sessie of op aanvraag
- **Routing**: Team Inbox → juiste Bucket via het team

### Tracing Rules

- Elke journal entry linkt via `[[wikilinks]]` terug naar de bron
- Elk deliverable bevat `Delivered on:` en `Delivered at:`
- Elk CRM-bestand bevat een Mentions log met links naar journal entries
